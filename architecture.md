# ARCHITECTURE.md — Архитектура сервиса кредитного скоринга

## 1. Архитектура

Для этого проекта выбран монолитный подход — весь сервис в одном контейнере

Микросервисы имеют смысл когда у компонентов разные требования к масштабированию. 
В нашем случае это учебный проект с одним доменом (кредитный скоринг) и двумя версиями одной модели
разбивать его на несколько сервисов означало бы добавить сложность без какой-либо выгоды.
Переходить на микросервисы стоит когда появятся реальные узкие места: например, если
препроцессинг данных станет тяжёлым и его нужно будет масштабировать отдельно от инференса.

| Критерий | Монолит (наш выбор) | Микросервисы |
|---|---|---|
| Развёртывание | Один `docker compose up` | Несколько сервисов, service discovery |
| Latency | Нет сетевых вызовов между компонентами | Каждый вызов — HTTP/gRPC |
| Отладка | Один лог, один контейнер | Нужен distributed tracing |
| Масштабирование | Несколько реплик контейнера | Независимое масштабирование компонентов |

## 2. Gunicorn + Nginx в production

В разработке Flask запускается через `python api.py` — это встроенный
однопоточный сервер, он не рассчитан на реальную нагрузку.

В production схема выглядит так:

Интернет → Nginx :80/:443 → Gunicorn workers :5010 → Flask → Модель

**Nginx** стоит перед всем остальным и берёт на себя:
- завершение TLS (HTTPS)
- буферизацию медленных клиентов
- rate limiting
- балансировку между репликами

**Gunicorn** запускает несколько worker-процессов Flask, что позволяет
обрабатывать несколько запросов параллельно и не использовать `app.run()` в production.

## 3. Логирование и мониторинг

Все логи сервиса пишутся в **stdout контейнера**. Их можно посмотреть через:

```bash
docker logs -f credit_default_api
docker logs -f credit_default_nginx
```

В нашем проекте Nginx логирует входящие HTTP-запросы, а Flask/Gunicorn
логируют работу самого API.

В production эти логи можно собирать через **ELK-стек**:

stdout → Filebeat → Logstash → Elasticsearch → Kibana

Это позволяет смотреть:
- количество запросов
- количество ошибок
- время ответа
- распределение предсказаний модели

Например, если резко выросло число ошибок или изменилась доля клиентов с
предсказанием дефолта, это может быть сигналом проблем с сервисом или data drift.

## 4. RabbitMQ

Сейчас наш API работает синхронно: клиент отправляет один запрос и сразу
получает один ответ.

Если в будущем появится задача обрабатывать большие файлы с тысячами клиентов,
то лучше делать это асинхронно через очередь задач.

Возможная схема:

Клиент → API → RabbitMQ → ML Workers → База / файл с результатами

RabbitMQ мог бы пригодиться для:
- батч-предсказаний по большому CSV-файлу
- фонового логирования предсказаний
- повторной обработки задач если воркер упал
- разгрузки основного API

То есть обычные одиночные запросы можно оставить синхронными, а тяжёлые задачи
перевести в очередь.

## 5. Конвертация модели в ONNX

Модель можно преобразовать в формат ONNX, чтобы ускорить инференс и сделать
запуск менее зависимым от версии Python и sklearn.

Плюсы ONNX:
- часто работает быстрее на CPU
- удобен для production-развёртывания
- проще переносить между разными окружениями

В нашем проекте это не реализовано, но при необходимости модель из sklearn
можно конвертировать в `.onnx`, а затем запускать через ONNX Runtime вместо
обычного `pickle`.

## 6. DVC и MLflow

**DVC** можно использовать для версионирования датасета и файлов моделей.
Это полезно, чтобы всегда понимать, на каких данных была обучена конкретная версия.

**MLflow** можно использовать для хранения результатов экспериментов:
- какие гиперпараметры были у модели
- какие метрики получились
- какая версия модели признана лучшей

В учебном проекте это не обязательно реализовывать полностью, но как концепт
MLOps это важно упомянуть.

## 7. Бизнес-метрики

Кроме ML-метрик, в таком проекте важны и бизнес-метрики.

**Expected Loss Reduction (ELR)** показывает, сколько денег банк может
сэкономить, если новая модель лучше обнаруживает дефолты.

**Approval Rate** показывает, какую долю заявок банк одобряет при выбранном
уровне риска.

Это важно потому что модель должна быть не только “точной” по F1-score, но и
полезной для бизнеса: снижать потери и не блокировать слишком много хороших клиентов.
